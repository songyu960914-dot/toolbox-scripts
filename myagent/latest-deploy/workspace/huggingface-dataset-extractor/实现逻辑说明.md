# HuggingFace 数据集元信息提取工具 - 实现逻辑说明

## 1. 整体流程

```
输入 Excel (桌面) → 逐行读取 URL → 调用 HuggingFace API → 提取元数据字段
                                   → 获取 README 内容
                                   → 获取数据预览（前几行）
                                   → LLM 判断榜单类型 + Agent 类型
                                   → 规则检测内容警告
                                   → 输出 Excel (桌面)
```

**输入格式：** 桌面 xlsx 文件，第一列为序号，第二列为 HuggingFace 数据集 URL。

**输出格式：** 桌面 xlsx 文件，包含 22 个字段。

---

## 2. 数据源

每个数据集从三个来源获取信息：

| 来源 | URL | 用途 |
|------|-----|------|
| HuggingFace API | `https://huggingface.co/api/datasets/{id}` | 元数据主体（tags, cardData, downloads 等） |
| README 原文 | `https://huggingface.co/datasets/{id}/raw/main/README.md` | 关键词提取、LLM 分析 |
| 数据预览 API | `https://datasets-server.huggingface.co/first-rows` | 列名和样本数据，辅助 LLM 判断 |

---

## 3. 字段提取逻辑

### 3.1 基础字段（直接从 API 获取）

| 字段 | 来源 | 说明 |
|------|------|------|
| 发布/更新时间 | `api.lastModified` | ISO格式转 YYYY-MM-DD |
| 下载量 | `api.downloads` | 直接取值 |
| 点赞量 | `api.likes` | 直接取值 |

### 3.2 双源合并字段

这些字段同时存在于 `api.tags` 数组和 `api.cardData` 对象中，需要从两处提取后去重合并。

**提取逻辑：**
1. 从 `tags` 数组中按前缀提取（如 `license:mit` → `mit`）
2. 从 `cardData` 对应字段提取（如 `cardData.license`）
3. 合并去重，保持顺序（大小写不敏感去重）

| 字段 | tags 前缀 | cardData 字段 |
|------|-----------|---------------|
| License | `license:` | `cardData.license` |
| Tasks | `task_categories:` | `cardData.task_categories` |
| 量级等级（条） | `size_categories:` | `cardData.size_categories` |
| 数据类型（文件类型） | `modality:` | `cardData.modality` |
| 数据格式 | `format:` | `cardData.format` |
| 语种 | `language:` | `cardData.language` |
| Tags | 无前缀的标签（不含 `:`） | `cardData.tags` |

### 3.3 计算字段

**数据大小（GB）：**
- 优先从 `cardData.download_size` 获取
- 若为空，遍历 `cardData.dataset_info` 中各配置的 `download_size` 求和
- 转换：`bytes / (1024^3)` 保留4位小数

**数据量级（条）：**
- 遍历 `cardData.dataset_info` 中所有 splits 的 `num_examples` 求和

**是否有论文：**
- 从 `tags` 中提取 `arxiv:` 前缀
- 从 `cardData.arxiv` 提取
- 合并后若有值 → "是"，拼接 arXiv URL

**是否有测试集：**
- 遍历 `dataset_info.splits`，检查是否有 name 包含 "test" 的 split

## 4. 榜单类型判断

### 4.1 分类标准（四选一）

| 类型 | 定义 |
|------|------|
| benchmark榜单 | 数据集本身就是 benchmark/测评基准，用于评测模型性能 |
| 名字含bench类 | 仓库名含 "bench" 但 README 为空或内容不足 |
| 其他榜单 | 包含评测/评估内容但不是标准 benchmark |
| 非榜单 | 训练数据，不用于评测 |

### 4.2 判断方式

通过 LLM 综合判断（见第6节合并调用）。LLM 会根据 README 内容、Tags、数据预览等信息判断。

**输出：**
- 榜单类型：四选一
- 榜单关键词原句：如果是 benchmark/其他榜单，提取关键证据原句（最多200字符）；否则为 "/"

---

## 5. Agent 类型判断

### 5.1 分类标准（六选一）

| 类型 | 定义 |
|------|------|
| 代码/机器人 | 任务目标是输出代码或控制机器人（代码+机器人算一个标签） |
| 多模态 | 每条数据都包含图片/音频/视频等多模态内容 |
| 通用 | 纯文本任务 |
| 混合可用 | 包含两种或以上不同类别，且有可用数据 |
| 混合不可用 | 代码/机器人 + 多模态组合，缺乏纯文本通用数据 |
| 其他 | 信息不足，无法判断 |
| / | 完全无法获取任何信息 |

### 5.2 特殊规则

- **代码 + 机器人不算混合**：代码类和机器人类归为同一标签 "代码/机器人"
- **tags 快速判断**：如果 tags 同时包含 robotics 相关标签和 multimodal 标签 → 直接判定为 "混合不可用"

### 5.3 混合说明

- 非混合类型：填 "/"
- 混合可用：注明具体组合（如 "通用+多模态"、"通用+代码/机器人"）
- 混合不可用：填 "代码/机器人+多模态"

## 6. LLM 合并调用

### 6.1 设计思路

v1 版本中榜单判断和 Agent 判断各调用一次 LLM（共2次/条），v2 合并为一次调用，节省成本和时间。

### 6.2 调用条件

当 README 为空且数据预览为空时，跳过 LLM 调用，直接返回默认值（非榜单、/、/、/）。

### 6.3 输入信息

发送给 LLM 的上下文包括：
- 数据集 ID
- Tasks、Modalities、Tags
- cardData 描述（前500字符）
- README 摘要（前1500字符）
- 数据预览（列名 + 前3行样本）

### 6.4 输出解析

要求 LLM 返回 JSON：
```json
{
  "榜单类型": "...",
  "榜单原句": "...",
  "Agent类型": "...",
  "混合说明": "..."
}
```

解析策略：
1. 优先从 `content` 中用正则提取 JSON
2. 如果失败，从 `reasoning_content` 中提取（某些模型将结果放在思维链中）
3. 正则匹配模式：`\{[^{}]*榜单[^{}]*Agent[^{}]*\}` 或含两个关键字段的 JSON 块
4. 对提取的值做模糊匹配（容忍 LLM 输出不完全标准的文本）

### 6.5 容错处理

- LLM 调用失败时：根据 tags 快速规则给出默认值
- JSON 解析失败时：保留默认值（非榜单 + 其他）
- LLM 返回非标准值时：模糊匹配到最接近的标准值

### 6.6 LLM 配置

- 模型：GLM-5-turbo（智谱）
- Temperature: 0.1（低随机性，提高一致性）
- Max tokens: 2000

## 7. 内容警告检测

### 7.1 检测逻辑（纯规则，不用 LLM）

**检查范围：**
- README 全文
- cardData 中的 description
- tags 中的特殊标签（nsfw, not-for-all-audiences, adult, 18+）

**匹配模式：**
```
(contain|include|有|包含) + 0~30字符 + (harmful|offensive|有害|冒犯)
(pornograph|sexually explicit|nsfw|色情|成人内容)
benchmark + 0~50字符 + (harmful|offensive|有害|冒犯)
```

### 7.2 排除规则

以下语句模式视为"作者呼吁/免责声明"，不算作真正的内容警告：
```
(should not|must not|do not|请勿|不应|禁止) + (use|使用|用于)
(responsibility|负责|responsible)
(recommend|建议|suggest)
```

### 7.3 输出

- 是否有警告：是/否
- 警告原因：匹配到的原句（最多3条，每条最多250字符），或 "/"

## 8. 网络请求策略

### 8.1 重试机制

所有 HTTP 请求（API、README、数据预览）均带重试：
- 最大重试次数：3（config.yaml 可配置）
- 重试间隔：递增（`retry_interval * (attempt + 1)` 秒）
- 仅对网络错误重试（ConnectionError、Timeout），HTTP 4xx/5xx 不重试

### 8.2 串行间隔控制

每条数据处理完后 sleep 1.5秒（可配置），避免触发 HuggingFace 和智谱的速率限制。

### 8.3 超时设置

- 请求超时：30秒
- 数据预览 info 接口：15秒（辅助接口，超时影响较小）

## 9. 版本演进

| 版本 | 特点 | 状态 |
|------|------|------|
| v1 | 串行 + 双 LLM 调用 + 硬编码配置 | 已废弃（编码损坏） |
| v2 | 串行 + 合并 LLM 调用 + 配置外部化 | **推荐使用** |
| v3 | 并发版（基于 v2 加了 ThreadPoolExecutor） | 仅供参考（并发易触发429） |

### v1 → v2 改进点
- 两次 LLM 调用合并为一次（成本减半）
- API Key 从硬编码改为 config.yaml 外部配置
- 支持环境变量占位符（`${ZHIPU_API_KEY}`）

### v2 → v3 改进点
- 加入多线程并发（ThreadPoolExecutor）
- 加入线程锁保护 print 输出
- 实际使用中发现并发会触发智谱 API 429 速率限制，效果不佳

## 11. 使用方式

```bash
cd huggingface-dataset-extractor/huggingface-dataset-extractor_v2
python run_workbook.py 工作簿1
```

- 输入文件放桌面，扩展名 `.xlsx`
- 输出文件自动保存到桌面，文件名含时间戳
- 本地 config.yaml 使用真实 API Key
- GitHub 上 config.yaml 使用 `${ZHIPU_API_KEY}` 占位符
