# HuggingFace 数据集元信息提取脚本 - 实现逻辑说明

## 1. 整体流程

```
用户输入 Excel 文件（A列序号、B列数据集URL）
    ↓
逐行读取，对每个数据集执行：
    1. 调用 HuggingFace API 获取元数据（tags、cardData、下载量等）
    2. 抓取 README.md 原文
    3. 从 tags 和 cardData 两处提取各字段，去重合并
    4. 计算数据大小、数据量级
    5. 判断榜单类型（关键词）
    6. 判断 Agent 类型（LLM 推理）
    ↓
输出 Excel 文件（包含17个元信息字段 + 榜单类型 + Agent类型）
```

---

## 2. 字段提取逻辑

### 2.1 基础字段（API 直接获取）
- **发布/更新时间**: `api_data.lastModified`，格式化为 YYYY-MM-DD
- **下载量**: `api_data.downloads`
- **点赞量**: `api_data.likes`

### 2.2 分类字段（从 tags + cardData 双源提取，去重合并）

#### License
- **来源1**: `tags` 数组中 `license:xxx` 格式的值
- **来源2**: `cardData.license`（可能是字符串或数组）
- **合并**: 去重后用逗号拼接

#### Tasks（任务类型）
- **来源1**: `tags` 中 `task_categories:xxx`
- **来源2**: `cardData.task_categories`
- **示例**: `text-generation`, `question-answering`, `code-generation`

#### 量级等级（条）
- **来源1**: `tags` 中 `size_categories:xxx`
- **来源2**: `cardData.size_categories`
- **示例**: `10K<n<100K`, `1M<n<10M`

#### 数据类型（文件类型 / modality）
- **来源1**: `tags` 中 `modality:xxx`
- **来源2**: `cardData.modality`
- **示例**: `text`, `image`, `audio`, `video`

#### 数据格式
- **来源1**: `tags` 中 `format:xxx`
- **来源2**: `cardData.format`
- **示例**: `parquet`, `json`, `csv`

#### 语种
- **来源1**: `tags` 中 `language:xxx`
- **来源2**: `cardData.language`
- **示例**: `en`, `zh`, `multilingual`

#### Tags（自定义标签）
- **来源1**: `tags` 数组中不含冒号的值（即非分类标签）
- **来源2**: `cardData.tags`
- **示例**: `sft`, `chatml`, `conversational`

### 2.3 规模字段（从 dataset_info 计算）

#### 数据大小（GB）
- **来源**: `cardData.download_size` 或遍历 `dataset_info[].download_size` 累加
- **计算**: 字节转 GB，保留4位小数
- **示例**: 0.0123 GB

#### 数据量级（条）
- **来源**: 遍历 `dataset_info[].splits[]`，累加所有 split 的 `num_examples`
- **示例**: 25000（表示25000条数据）

### 2.4 论文字段

#### 是否有论文
- **来源**: `tags` 中 `arxiv:xxx` 或 `cardData.arxiv`
- **规则**: 有 arxiv ID 则为"是"，否则为"否"

#### 论文arXivURL
- **构建**: `https://arxiv.org/abs/{arxiv_id}`，多篇用逗号拼接
- **示例**: `https://arxiv.org/abs/2307.09288`

### 2.5 测试集字段

#### 是否有测试集
- **来源**: 遍历 `dataset_info[].splits[]`，检查是否有 `name` 含 `test` 的 split
- **补充**: 也检查 `cardData.configs[].data_files[]` 中是否有 `split='test'`
- **规则**: 有 test split 则为"是"，否则为 `/`（表示无或未知）

### 2.6 内容安全字段

#### 是否有警告标签
- **检测范围**: README、cardData.description、tags
- **检测内容**: **仅检测数据集中确实存在的有害、色情内容**
- **排除内容**: 作者的呼吁式警告（如"请勿用于..."、"不应用于..."、"建议..."、"负责任使用"等）

**检测规则：**

1. **NSFW/成人内容标签**
   - Tags: `not-for-all-audiences`, `nsfw`, `adult`, `18+`
   - 检测到 → 标记"标签含: nsfw"

2. **数据集包含有害内容的关键模式**
   - 正向匹配：
     - `(contain|include|有|包含).{0,30}(harmful|offensive|有害|冒犯)`
     - `pornograph|sexually explicit|nsfw|色情|成人内容`
     - `benchmark.{0,50}(harmful|offensive|有害|冒犯)`
   - 排除模式（呼吁式）：
     - `(should not|must not|do not|请勿|不应|禁止).{0,30}(use|使用|用于)`
     - `(responsibility|负责|responsible|recommend|建议|suggest)`

3. **提取原句**
   - 从 README 中提取关键词所在原句（最多250字符）
   - 最多返回3句
   - 去重

**输出格式：**
- `是否有警告`: "是" / "否"
- `警告原因`: 
  - 无警告：`/`
  - 有警告：标签信息 或 原文句子，多个用分号分隔
  - 示例：`标签含: nsfw; This benchmark contains content that may be considered harmful or offensive`

## 3. 榜单类型判断逻辑

### 3.1 判断方式：关键词匹配 + LLM 二次确认

**第一阶段：关键词初筛**

合并以下文本来源（全部转小写，**仅 README，不含数据预览**）：
- README.md 原文
- `tags` 数组
- `cardData.description`

### 3.2 分类规则（按优先级）

#### 优先级1: **benchmark榜单**
- **关键词**: `bench`, `benchmark`, `longbench`
- **匹配位置**: README、tags、description 任一处包含即判定
- **示例**: MMLU、GSM8K、HumanEval、LongBench-Chat

#### 优先级2: **名字含bench类**
- **触发条件**:
  1. 仓库名（dataset_id）含 `bench` 或 `benchmark`
  2. **且** README 为空或过短（去除 YAML front matter 后 <50字符）
- **用途**: 捕捉仓库名暗示是 benchmark 但文档缺失的情况
- **示例**: `user/minibench`（README 为空）

#### 优先级3: **其他榜单**
- **中文关键词**: `测试`, `测评`, `评估`, `评价`, `基准`
- **英文关键词**: `evaluation`, `evaluate`, `eval`, `assessment`, `test set`, `leaderboard`, `scoring`, `metric`
- **用途**: 包含评测内容但不是标准 benchmark（中英文均适用）

#### 默认: **非榜单**
- 以上都不符合时判定为训练数据集

**第二阶段：LLM 二次确认**

当初步判断为 `benchmark榜单` 或 `其他榜单` 时：
1. 提取关键词所在原句（从 README 中）
2. 将原句和上下文交给 LLM 分析
3. LLM 判断：关键词在语境中是否真的表达"该数据集用于测评/评估"的含义
4. 如果只是顺带提及（如"我们在 benchmark 上测试了效果"），改判为"非榜单"

**输出字段：**
- `榜单类型`: benchmark榜单 / 名字含bench类 / 其他榜单 / 非榜单
- `榜单关键词原句`: 关键词所在的原文句子（最多300字符）

## 4. Agent 类型判断逻辑

### 4.1 判断方式：LLM 推理（GLM-5-turbo）+ 规则辅助

#### 输入内容：
- 数据集 ID
- Tasks、Modalities、Tags（元数据）
- cardData 描述（前500字符）
- README 摘要（前1500字符）
- **数据预览**：列名 + 前3行样本值（实际数据内容）

#### 数据预览获取：
- API: `https://datasets-server.huggingface.co/first-rows`
- 优先尝试 `config=default, split=train`
- 失败时自动获取可用的 config 和 split
- 提取：列名（最多15列）+ 前5行数据（每列截断至200字符）

#### 特殊规则（优先于 LLM）：
1. **README 和数据预览都为空** → 直接返回 `/`
2. **Tags 含 `robotics` + `multimodal`** → 直接返回 `混合不可用`（代码/机器人+多模态）

#### Prompt 设计（简化版）：
```
根据以下数据集信息判断Agent类型（六选一）：

- 代码/机器人: 任务目标要求输出代码或控制机器人。注意：代码和机器人算一个标签
- 多模态: 每条数据都包含图片/音频/视频（如每行都有screenshot列）
- 通用: 纯文本任务（对话、问答、推理、浏览器操作等），不要求输出代码
- 混合可用: 包含两种或以上不同类别，且有可用数据
- 混合不可用: 代码/机器人 + 多模态，缺乏通用数据
- 其他: 信息不足
- /: 无法获取信息

混合判断规则：
1. 代码 + 机器人 → 不算混合，归为"代码/机器人"
2. 通用 + 多模态 → 混合可用，说明"通用+多模态"
3. 通用 + 代码/机器人 → 混合可用，说明"通用+代码/机器人"
4. 代码/机器人 + 多模态 + 通用 → 混合可用，说明"通用+代码/机器人+多模态"
5. 代码/机器人 + 多模态（无通用） → 混合不可用，说明"代码/机器人+多模态"

关键判断依据：
1. 优先看数据预览中的列名和样本值
2. 如果每行都有 screenshot/image 列 → 多模态（不是混合）
3. 如果部分数据是纯文本、部分是图片/代码 → 判断具体混合类型

返回 JSON: {"Agent类型":"...", "混合说明":"..."}

混合说明填写规则：
- 非混合类型：填 "/"
- 混合类型：注明具体组合，如"通用+多模态"、"代码/机器人+多模态"
```

#### 响应解析：
1. GLM-5-turbo 是推理模型，答案在 `reasoning_content` 字段
2. 用正则提取 JSON: `\{[^{}]*Agent[^{}]*\}`
3. 模糊匹配到六个标准值之一
4. 提取 `混合说明` 字段
5. 非混合类型自动将混合说明改为 `/`
6. 默认返回 `("其他", "/")`

### 4.2 Fallback 机制

若 LLM 调用失败（网络错误、JSON 解析失败等），fallback 到：
- 直接返回 **("其他", "/")**

### 4.3 六分类定义

#### 代码/机器人
- **核心特征**: 任务目标要求输出可执行代码，或控制机器人
- **关键词示例**: code-generation, code-completion, text-to-code, robotics, embodied
- **数据示例**: HumanEval、MBPP、代码对话数据、机器人操作轨迹
- **重要**: 代码类和机器人类算一个标签，不构成混合

#### 多模态
- **核心特征**: 每条数据都包含非文本模态（图片、音频、视频）
- **判断依据**: 
  - 数据预览中**每行都有** `screenshot/image/audio` 等列
  - 优先看 `modality` 字段是否含 `image/audio/video/3d`
- **数据示例**: COCO、AudioCaps、视觉问答数据集、Multimodal-Mind2Web

#### 通用
- **核心特征**: 纯文本任务，不要求输出代码
- **包含**: 对话、问答、推理、摘要、翻译、工具调用、浏览器操作、函数调用
- **关键**: 工具调用(tool-calling)、浏览器agent等如果输出的是自然语言指令（如"click button"），归这一类
- **数据示例**: Alpaca、ShareGPT、BrowserAgent-Data

#### 混合可用
- **触发条件**: 数据集包含两种或以上不同类别（代码/机器人、多模态、通用中至少两种）
- **判断依据**: 数据集中有不同类型的任务或数据，且存在可用的纯文本或通用数据
- **混合组合**:
  - 通用 + 多模态（例如：对话数据 + 图像理解任务）
  - 通用 + 代码/机器人（例如：对话数据 + 代码生成任务）
  - 通用 + 代码/机器人 + 多模态（全包含且有通用数据）
- **输出格式**: 
  - `Agent类型`: 混合可用
  - `混合说明`: 注明具体组合，如"通用+多模态"、"通用+代码/机器人+多模态"

#### 混合不可用
- **触发条件**: 代码/机器人 + 多模态，但缺乏纯文本通用数据
- **用途**: 机器人视觉控制数据集（操作指令 + 图像观测，但不含对话/问答等通用数据）
- **数据示例**: InternRobotics/InternData-A1（机器人操作轨迹 + 相机图像）、SimWorld-AI/SimWorld（机器人仿真 + 多模态观测）
- **输出格式**:
  - `Agent类型`: 混合不可用
  - `混合说明`: "代码/机器人+多模态"

#### 其他
- **触发条件**: 信息不足以判断具体类别
- **用途**: README 和数据预览都缺失或模糊，LLM 无法确定分类
- **输出格式**:
  - `Agent类型`: 其他
  - `混合说明`: /

#### /（斜杠）
- **触发条件**: README 和数据预览**都为空**
- **用途**: 完全无法获取任何有效信息
- **输出格式**:
  - `Agent类型`: /
  - `混合说明`: /

## 5. 容错处理

### 5.1 API 请求容错
- HTTP 超时: 30秒
- README 抓取重试: 最多3次，间隔递增（2s、4s、6s）
- 连接被重置时: 记录错误，该数据集字段填 `/`

### 5.2 数据类型容错
- `cardData` 可能是字符串（非预期）→ 转为空字典
- `dataset_info` 可能是 dict 或 list → 统一处理为 list
- README 可能返回空或非 200 → 默认空字符串

### 5.3 缺失值处理
- 数值字段无数据时填 `/`
- 字符串字段无数据时填 `/`
- boolean 字段："是"、"否"、或 `/`

## 6. 性能与成本

### 6.1 运行速度
- 单条数据集: ~5-7秒（1s API + 1s README + 2s 数据预览 + 2s LLM（含榜单+Agent两次调用）+ 1.5s 间隔）
- 25条: 约3-4分钟
- 100条: 约10-12分钟

### 6.2 API 成本（GLM-5-turbo）
- **榜单二次确认**: ~800 tokens/次（仅命中 benchmark/其他榜单时调用）
- **Agent判断**: ~1500-2000 tokens/次（含数据预览）
- **总输入**: ~1200-2500 tokens/条（取决于是否需要榜单确认）
- **输出（含推理）**: ~400-600 tokens/条
- 定价: ¥0.005/千tokens（输入）+ ¥0.01/千tokens（输出）
- **25条成本**: 约 ¥0.4-0.6
- **100条成本**: 约 ¥1.6-2.4

### 6.3 限流控制
- 每条处理后 sleep 1.5秒（避免触发 HuggingFace 限流）
- LLM 请求 temperature=0.1（确保输出稳定性）
- 数据预览获取有重试机制（最多3次，间隔递增）
- README 获取有重试机制（最多3次，间隔递增）

## 10. 已知限制

1. **数据预览依赖 datasets-server**: 部分数据集可能未启用 datasets-server（会跳过数据预览，LLM 仅基于 README 判断）
2. **Agent 类型边界模糊**: 工具调用类数据集（如 tool-calling、function-calling）归类存在争议
3. **单线程执行**: 未使用并发，大批量处理较慢（可改进）
4. **错误重试有限**: README/数据预览抓取失败3次后放弃，可能丢失部分信息
5. **数据预览可能不全**: 仅取前5行数据，无法覆盖所有 category 类型（混合数据集可能漏判）

---

## 11. 后续改进方向

1. **并发处理**: 使用 asyncio 或多线程加速（可提速3-5倍）
2. **缓存机制**: 对已处理的数据集缓存元数据和数据预览，避免重复请求
3. **增量更新**: 支持只处理新增/变更的数据集
4. **配置外部化**: API Key、模型选择等通过配置文件管理
5. **增强内容安全检测**: 使用 LLM 分析 README 全文，识别隐性风险内容
6. **数据预览扩展**: 增加采样行数，或使用 datasets-server 的统计 API 获取列值分布

## 12. 内容安全判断逻辑

### 12.1 检测维度

**核心原则：只检测数据集中确实存在的有害、色情内容，排除作者的呼吁式警告**

#### 1. NSFW / 成人内容标签
- **Tags 检查**: `not-for-all-audiences`, `nsfw`, `adult`, `18+`
- **标记**: 检测到 → 警告原因添加"标签含: nsfw"（或其他具体标签名）

#### 2. 数据集包含有害内容的关键模式
- **正向匹配模式**（描述数据集包含有害内容）：
  - `(contain|include|有|包含).{0,30}(harmful|offensive|有害|冒犯)`
  - `(pornograph|sexually explicit|nsfw|色情|成人内容)`
  - `benchmark.{0,50}(harmful|offensive|有害|冒犯)`

- **排除模式**（呼吁式语句，不算警告）：
  - `(should not|must not|do not|请勿|不应|禁止).{0,30}(use|使用|用于)`
  - `(responsibility|负责|responsible)`
  - `(recommend|建议|suggest)`

- **逻辑**：
  1. 在 README/description 中查找正向匹配模式
  2. 提取匹配的原句
  3. 检查原句是否包含排除模式
  4. 如果包含排除模式 → 不标记为警告（是作者呼吁）
  5. 如果不包含排除模式 → 标记为警告（数据集确实有问题）

**示例：**
- ✅ 标记为警告："This benchmark contains content that may be considered harmful or offensive"
- ❌ 不标记："Users should not use this dataset for harmful purposes"（呼吁式）
- ❌ 不标记："We recommend responsible use of this data"（呼吁式）

### 12.2 判断优先级

1. **强警告**: NSFW 标签 → 必标记
2. **内容警告**: 正向匹配且不是呼吁式 → 标记
3. **无警告**: 未匹配任何模式或仅包含呼吁式语句 → 不标记

### 12.3 输出格式

| 是否有警告 | 警告原因 |
|-----------|----------|
| 否 | / |
| 是 | 标签含: nsfw |
| 是 | This benchmark contains content that may be considered harmful or offensive |
| 是 | 标签含: adult; Pornographic images included in the dataset |

repo_name = dataset_id.lower()

    return 'benchmark榜单'
elif name_has_bench and readme_empty:
    return '名字含bench类'
elif '测试' in all_text or '评估' in all_text:
    return '其他榜单'
else:

```python
def check_content_warning(api_data, tags, readme_text, card_data):
    warnings = []
    
    # 1. Gated dataset
    if api_data.get('gated'):
        warnings.append('需申请访问')
    
    # 2. NSFW 标签
    nsfw_tags = {'not-for-all-audiences', 'nsfw', 'adult', '18+'}
    if any(tag in tags for tag in nsfw_tags):
        warnings.append('NSFW/成人内容')
    
    # 3. 敏感内容关键词
    all_text = (readme_text + ' ' + str(tags) + ' ' + str(card_data.get('description', ''))).lower()
    sensitive_keywords = [
        'warning', 'harmful', 'offensive', 'sensitive', 'explicit',
        'violence', 'disturbing', 'toxic', 'bias', 'discriminatory',
        '警告', '有害', '冒犯', '敏感', '暴力', '歧视'
    ]
    if any(kw in all_text for kw in sensitive_keywords):
        warnings.append('含敏感内容')
    
    # 4. 高风险任务
    risk_tasks = {'hate-speech-detection', 'toxicity-detection'}
    task_tags = [t.split(':')[1] for t in tags if t.startswith('task_categories:')]
    if any(task in task_tags for task in risk_tasks):
        warnings.append('含风险任务数据')
    
    if warnings:
        return '是', ', '.join(list(dict.fromkeys(warnings)))  # 去重
    return '否', '/'
```
